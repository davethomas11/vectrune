# Growth Backlog User Stories

## Onboarding

- As a new user, I want one obvious example to start with so I can get a useful result quickly.
- As a new user, I want install and run steps to work as written so I can trust the project quickly.
- As a new user, I want the first docs page to tell me what Vectrune is good for so I know whether to keep investing time.
- As a new user, I want examples to be labeled by difficulty and use case so I do not have to guess where to begin.

## Trust

- As an evaluator, I want generated output and runtime behavior to be easy to inspect so I can understand what the tool is doing.
- As an evaluator, I want safe defaults and predictable behavior so I can recommend Vectrune to a team.
- As an evaluator, I want docs to clearly mark current limits so I do not overestimate what is supported.

## Copy-paste workflows

- As a builder, I want starter examples that look like real projects so I can reuse them instead of rewriting from scratch.
- As a builder, I want short recipe-style docs for common tasks so I can adapt them into internal tools and prototypes.
- As a builder, I want examples to include expected outcomes so I can confirm I am using the feature correctly.

## Distribution and environment fit

- As a developer, I want `.rune` files to look good in my editor so learning the language feels less painful.
- As a developer, I want Windows, macOS, and Linux setup guidance to be clear enough that local environment issues do not block evaluation.
- As a developer, I want examples and tooling to work in common editor workflows so the language feels practical.

## Feedback loops

- As a maintainer, I want onboarding friction to turn into backlog items quickly so the product gets easier to adopt over time.
- As a maintainer, I want docs, examples, and runtime behavior to stay aligned so users do not lose trust.
- As a maintainer, I want a small number of polished adoption surfaces so the team can improve them consistently.
